## 4. 各步骤要点（框架）

> 本次只展开第 1 步；其余小节先占位，开发到对应步骤时再补设计要点与验收标准。

### 步骤 1：初始化 Hardhat 工程（`scaffold-hardhat-project`）✅ 已展开

详见对应 OpenSpec change：`openspec/changes/scaffold-hardhat-project/`。

要点：

- 初始化 Hardhat 工程与 npm 依赖（Hardhat、ethers v6、OpenZeppelin、测试库）。
- 配置 `hardhat.config`：Solidity 版本（与 OpenZeppelin 兼容，如 `0.8.20+`）、Sepolia 网络、编译优化。
- 建立目录骨架：`contracts/`、`scripts/`、`test/`，放置占位文件保证 `compile`/`test` 可跑通。
- 用 `.env.example` 约定 RPC URL 与私钥变量名，**不提交真实密钥**（见 [[项目讨论结果-去中心化二手交易担保托管]] 全局约束）。
- 验收：`npx hardhat compile` 与 `npx hardhat test` 在空骨架上成功执行。

### 步骤 2：核心担保托管合约（`add-escrow-core-contract`）✅ 已完成

对应方案：第 8 节状态机、第 9 节核心接口、13.4 合约安全、13.11.1/13.11.5/13.11.6 输入与冻结校验。

详见对应 OpenSpec change：`openspec/changes/add-escrow-core-contract/`。

要点：

- 新增核心合约 `EscrowMarketplace`，实现商品创建、改价、下架、付款托管、卖家标记交付、买家确认收货、卖家 pull 提款。
- 第 2 步仅覆盖基础闭环，状态使用 `Created`、`Locked`、`Delivered`、`DisputeDepositPending`、`Disputed`、`Inactive`；其中纠纷相关状态先预留，不实现进入和裁决逻辑。
- 商品在买家付款后冻结核心交易信息，防止卖家在资金托管后修改价格、描述、交付窗口等关键字段。
- 买家付款金额必须等于商品价格；卖家不能购买自己的商品；下架后不能购买；已购买商品不能重复购买。
- 卖家标记交付后，买家才能确认收货；买家确认后主交易结束，资金记入卖家的可提现余额。
- 资金释放使用 pull payment：先更新状态和余额，再由卖家单独提款，避免直接在确认函数中外部转账。
- 合约需使用重入保护，并为商品创建、改价、下架、付款、交付、确认收货、提款等关键动作发出事件。
- 验收：核心合约可编译，核心主流程与权限/状态/金额边界有单元测试覆盖。

结果摘要（2026-06-28）：

- 合约文件：`contracts/EscrowMarketplace.sol`（Solidity `^0.8.20`，OpenZeppelin `ReentrancyGuard`）。
- 测试文件：`test/EscrowMarketplace.test.js`（32 个测试用例）。
- 验证：`npx hardhat compile` 成功；`npx hardhat test` 33 passing。
- 未实现：退款、超时、仲裁、纠纷押金、前端、Sepolia 部署（留给后续步骤）。

### 步骤 5：多签仲裁投票（`add-arbitration-voting`）⬜ 待展开

对应方案：第 5/6 节仲裁与投票、13.1 超时兜底、13.2/13.7 纠纷押金奖励与证据哈希、14.1.3/14.2.1 兜底方向与纠纷押金去向。

已确认的押金奖励设计：

- 用固定 `disputeDeposit` 完全替代原 `arbitrationFee`，Demo 建议 `0.001 ETH`。
- 买家或卖家都可发起纠纷；买家发起默认诉求为退款，卖家发起默认诉求为放款。
- 发起方调用 `openDispute(itemId, evidenceHash)` 并支付 `disputeDeposit` 后，状态进入 `DisputeDepositPending`。
- 另一方需在 `disputeDepositWindow` 内调用 `respondDispute(itemId)` 补交同额押金；超时未补交则视为放弃抗辩，发起方自动胜诉，发起方押金退回，不发评审奖励。
- 双方都交押金后状态进入 `Disputed`，仲裁员才能投票。
- 达成 2/3 多数票时，胜诉方押金退回，败诉方押金平分给投多数票的仲裁员；少数票和未投票仲裁员不拿奖励。
- 仲裁超时未达 2/3 时，主交易资金按兜底规则结算，双方纠纷押金都退回，不发评审奖励。

状态机需在第 2 步核心合约阶段预留：

```solidity
enum State {
    Created,
    Locked,
    Delivered,
    DisputeDepositPending,
    Disputed,
    Inactive
}
```
