要点：

- 新增核心合约 `EscrowMarketplace`，实现商品创建、改价、下架、付款托管、卖家标记交付、买家确认收货、卖家 pull 提款。
- 第 2 步仅覆盖基础闭环，状态使用 `Created`、`Locked`、`Delivered`、`DisputeDepositPending`、`Disputed`、`Inactive`；其中纠纷相关状态先预留，不实现进入和裁决逻辑。

- 合约需使用重入保护，并为商品创建、改价、下架、付款、交付、确认收货、提款等关键动作发出事件。
- 验收：核心合约可编译，核心主流程与权限/状态/金额边界有单元测试覆盖。

### 步骤 3：退款与超时处理（`add-refund-and-timeout`）✅ 已完成

对应方案：第 4 节双向超时、第 5 节双方确认退款、14.1.1/14.1.2 超时放款须交付与计时器串联。

本步骤目标：

- 在第 2 步核心合约基础上，补齐“卖家不交付”和“买家拖延确认”两条超时链路。
- 实现双方确认退款的最小闭环：买家发起退款、卖家同意退款、托管资金退回买家。
- 明确两个计时窗是串行而非并行：
  - `deliveryWindow` 从买家付款进入 `Locked` 后开始计时；
  - `confirmWindow` 只在卖家 `markDelivered()` 后开始计时。
- 超时放款必须以“卖家已声明交付”为前提，防止卖家未发货却靠买家离线拿到资金。

建议本步在合约中补充：

- 时间字段：
  - `paidAt`：买家付款时间；
  - 继续使用现有 `deliveredAt`：卖家标记交付时间；
- 配置项：
  - `deliveryWindow`
  - `confirmWindow`
- 退款标记：
  - `refundRequested`
  - `refundRequestedAt`

建议新增函数：

- `requestRefund(uint256 itemId)`：
  - 由买家调用；
  - 适用于 `Locked` 或 `Delivered` 状态；
  - 仅做“发起退款申请”标记，不直接退款；
  - 发起后应阻止卖家再走 `releaseAfterTimeout()`。
- `approveRefund(uint256 itemId)`：
  - 由卖家调用；
  - 仅在买家已发起退款时可执行；
  - 将托管资金记账到买家的可提现余额；
  - 状态转为 `Inactive`。
- `releaseAfterTimeout(uint256 itemId)`：
  - 由卖家调用；
  - 仅在 `Delivered` 状态且 `block.timestamp >= deliveredAt + confirmWindow` 时可执行；
  - 必须校验未存在退款申请；
  - 将托管资金记账到卖家可提现余额；
  - 状态转为 `Inactive`。

本步边界：

- 不引入 `DisputeDepositPending -> Disputed` 的纠纷押金和仲裁流程。
- 不实现 `openDispute`、`respondDispute`、`submitEvidence`、`voteDispute`。
- 不处理仲裁员保证金、投票奖励、卖家保证金和信誉。
- 若卖家超出 `deliveryWindow` 未交付，`markDelivered` 必须 revert；买家仍可通过退款路径退出；升级纠纷留到步骤 5。
- 买家发起退款后，`confirmReceived` 与 `releaseAfterTimeout` 均不可执行，直至卖家 `approveRefund` 完成退款。

验收重点：

- 买家付款后卖家若长期不交付，买家可发起退款并由卖家同意完成退款。
- 卖家已交付后，买家若长时间不确认且未发起退款，卖家可超时放款。
- 卖家未交付时，不能直接走超时放款。
- 退款申请一旦存在，卖家不能再利用确认超时拿钱。
- 相关行为有单元测试覆盖，并更新 [[开发记录]]。

结果摘要（2026-06-28）：

- 合约文件：`contracts/EscrowMarketplace.sol` — 新增 `paidAt`、`refundRequested`、`refundRequestedAt` 字段；构造函数注入 `deliveryWindow` / `confirmWindow`（immutable）；新增 `requestRefund`、`approveRefund`、`releaseAfterTimeout` 及 3 个事件、5 个自定义 error；`markDelivered` 在 `paidAt + deliveryWindow` 内才可调用；`confirmReceived` 在 `refundRequested == true` 时 revert。
- 测试文件：`test/EscrowMarketplace.test.js` — 新增 refund/timeout 与交付窗口、退款阻塞确认收货测试（共 58 个 EscrowMarketplace 测试）。
- 验证：`npx hardhat compile` 成功；`npx hardhat test` 59 passing（58 EscrowMarketplace + 1 Placeholder）。

<summary>原设计要点（归档）</summary>

本步骤目标：

- 在第 4 步仲裁员质押基础上，补齐完整纠纷处理闭环。
- 买家或卖家都能发起纠纷，并用 `evidenceHash` 固定链下证据。
- 发起方先交固定 `disputeDeposit`，另一方在 `disputeDepositWindow` 内补交同额押金。
- 双方押金到位后进入 `Disputed`，仅有效且非当事人的仲裁员可以投票。
- 达到 2/3 多数票后自动执行裁决：支持买家则退款，支持卖家则放款。
- 正常裁决时，胜诉方纠纷押金退回，败诉方押金作为奖励平均分给投多数票的仲裁员。
- 仲裁押金补交超时或仲裁投票超时时，按已确认的兜底规则结算，避免资金永久锁死。

已确认的押金奖励设计：

- 用固定 `disputeDeposit` 完全替代原 `arbitrationFee`，Demo 建议 `0.001 ETH`。
- 买家或卖家都可发起纠纷；买家发起默认诉求为退款，卖家发起默认诉求为放款。
- 发起方调用 `openDispute(itemId, evidenceHash)` 并支付 `disputeDeposit` 后，状态进入 `DisputeDepositPending`。
- 另一方需在 `disputeDepositWindow` 内调用 `respondDispute(itemId)` 补交同额押金；超时未补交则视为放弃抗辩，发起方自动胜诉，发起方押金退回，不发评审奖励。
- 双方都交押金后状态进入 `Disputed`，仲裁员才能投票。
- 达成 2/3 多数票时，胜诉方押金退回，败诉方押金平分给投多数票的仲裁员；少数票和未投票仲裁员不拿奖励。
- 仲裁超时未达 2/3 时，主交易资金按兜底规则结算，双方纠纷押金都退回，不发评审奖励。

状态机：

```solidity
enum State {
    Created,
    Locked,
    Delivered,
    DisputeDepositPending,
    Disputed,
    Inactive
}
```

本步边界：

- 不实现卖家保证金、举报机制和信誉计数。
- 不实现前端、部署脚本或 Sepolia 部署记录。
- 不把证据原文上链，只保存 `bytes32 evidenceHash`。
- 不解决仲裁员女巫攻击，只在报告和后续文档中主动说明其治理边界。

### 步骤 6：卖家保证金/举报/信誉（`add-seller-stake-reputation`）✅ 已完成

对应方案：7.4 卖家保证金、7.5/13.2.1/13.2.2 举报与反向保护、7.6/13.9 信誉雏形。

本步骤目标：

- 让卖家在创建商品时支付固定 `sellerStakeAmount`，避免“买家锁钱、仲裁员质押、卖家零成本”的不对称。
- 正常成交、卖家超时放款、双方协商退款、付款前下架时退还卖家保证金。
- 纠纷裁决或兜底结算明确支持买家退款时，罚没卖家保证金，并记录卖家罚没次数。
- 建立轻量信誉计数：完成次数、纠纷次数、买家胜诉次数、卖家胜诉次数、卖家保证金罚没次数。
- 建立举报押金 `reportDeposit`：举报是“标记与复核入口”，不是自动惩罚仲裁员。
- “不投票”类举报由合约自动判断链上事实；“作恶”类举报保存证据哈希，由 owner 手动复核。

关键接口：

```solidity
createItem(bytes32 metadataHash, uint256 price) payable
reportNoVote(uint256 itemId, address arbiter) payable
reportMisconduct(uint256 itemId, address arbiter, bytes32 evidenceHash) payable
resolveReport(uint256 reportId, bool upheld)
```

建议新增配置与字段：

- `sellerStakeAmount`：卖家创建商品时必须支付的固定保证金，Demo 可用 `0.001 ETH`。
- `reportDeposit`：举报押金，Demo 可用 `0.0005 ETH`。
- 商品维度记录 `sellerStake`、`sellerStakeSettled`、`sellerStakeSlashed`。
- 地址维度记录 `completedTradeCount`、`disputeCount`、`buyerWinCount`、`sellerWinCount`、`sellerStakeSlashedCount`。
- 举报维度记录 `reportId`、`itemId`、`reporter`、`reported`、`reportType`、`status`、`evidenceHash`、`deposit`。

本步边界：

- 不把普通协商退款视为卖家作恶；卖家同意退款时保证金退还卖家。
- 不因仲裁员投少数票直接惩罚；投票结果是主观判断，不能和可证作恶混为一谈。
- 不自动罚没仲裁员保证金；举报押金只在举报人和被举报仲裁员之间按结果记账。
- 不实现复核陪审团、DAO 治理、稳定币计价、前端或 Sepolia 部署。

验收重点：

- 创建商品必须支付卖家保证金，金额错误应 revert。
- 正常完成、退款、下架、超时放款都只能结算卖家保证金一次。
- 仲裁支持买家时，卖家保证金被罚没并更新信誉计数。
- 仲裁支持卖家时，卖家保证金退还卖家。
- `reportNoVote` 能自动区分未投票和已投票仲裁员，并按结果处理举报押金。
- `reportMisconduct` 能创建带证据哈希的待复核举报，且只有 owner 能 `resolveReport`。
- 所有资金路径继续使用 `pendingWithdrawals` pull payment，不在结算函数中直接转账。

已知剩余风险（code review 回退后，2026-06-28 记录）：

1. **举报目标时效**：`_isValidReportTarget` 按举报时刻的 `activeArbiters` / `arbiterStakes` 判断。纠纷结束后已退出质押的仲裁员不可再被举报，即使其在纠纷期间未投票。若需纠纷入场时冻结可举报名单，须单独 OpenSpec change，且不得用 `respondDispute` 内无界列表遍历（gas DoS）。
2. **新质押仲裁员可被举报**：纠纷结束后新质押地址在现有语义下仍是合法举报目标，对 no-vote 场景语义偏弱；与原始合约设计一致，报告/PPT 中应说明为已知局限。
3. **投票门槛快照 ≠ 举报快照**：`arbiterCountSnapshot` / `voteThresholdSnapshot` 来自 `fix-dispute-vote-threshold-snapshot`，解决投票门槛被实时人数压低问题；举报可举报名单是独立问题，不可复用无界 `activeArbiterList` 方案。
