# 项目讨论结果：去中心化二手交易担保托管平台

> 讨论日期：2026-06-23
> 项目方向：Decentralized Escrow System
> 推荐场景：校园二手交易担保平台

---

## 1. 项目定位

本项目不是要完整复刻闲鱼、转转这类二手交易平台，而是做一个面向二手交易的 **链上担保支付层**。

传统二手平台主要解决商品展示、聊天、推荐、物流和客服问题；本项目重点解决的是交易中最关键的 **资金信任问题**：

- 买家怕先付款后卖家不发货。
- 卖家怕先发货后买家不付款。
- 双方都不希望完全依赖中心化平台后台。
- 纠纷处理希望更加公开、可追溯。

项目核心价值：

> 买家的资金先进入智能合约托管，卖家不能提前拿走；买家确认收货后，合约自动放款。如果发生退款纠纷，则通过多签仲裁和保证金机制处理。

## 2. 推荐实现场景

### 场景：校园二手交易

学生 A 想出售一台二手显示器，学生 B 想购买。双方可以通过微信群、校园论坛或线下方式沟通，但资金流转交给智能合约处理。

### 正常交易流程

1. 卖家发布商品，例如“二手显示器”，价格为 `0.01 ETH`，此时状态为 `Created`。
2. 付款前，卖家可以在 `Created` 状态下修改名称、描述、价格，或直接下架（`cancelItem`）。
3. 买家连接 MetaMask，点击购买，支付的 ETH 进入智能合约托管。
4. 合约状态从 `Created` 变为 `Locked`，记录付款时间 `paidAt`，**商品信息从此冻结，不能再改**。
5. 卖家线下交付商品后，调用 `markDelivered()` 在链上声明已交付，状态从 `Locked` 变为 `Delivered`，**确认收货倒计时 `confirmWindow` 从这一刻才开始计时**。
6. 买家收到商品后点击确认收货 `confirmReceived`。
7. 合约把托管资金记账到卖家名下（pull 提款模式），状态变为 `Inactive`，发出 `Released` 事件。
8. 卖家自己调用 `withdraw()` 取走资金。

> 资金不直接 `transfer` 给卖家，而是先记到 `pendingWithdrawals`，由收款方主动提取，避免对方是会 revert 的合约地址时卡死整笔交易（见第 7 节与第 13.4 节）。

### 链上和链下分工

链下负责：

- 商品图片
- 商品描述
- 买卖双方沟通
- 线下交付
- 纠纷证据原文，例如照片、聊天记录、物流信息（链上只存其哈希）

链上负责：

- 卖家地址、买家地址
- 商品价格、交易状态、付款时间 `paidAt`
- 卖家交付声明 `delivered`
- 托管资金、放款 / 退款 / 超时结算逻辑（pull 提款）
- 仲裁投票、仲裁超时兜底、仲裁费记账
- 仲裁员保证金、卖家保证金记录
- 证据哈希（买卖双方各一份）
- 举报记录与判定
- 买卖双方信誉计数（完成 / 纠纷次数）
- 覆盖全流程的事件日志

## 3. 和传统平台相比的优势

本项目不强调商品展示和聊天体验超过传统平台，而是强调资金托管和纠纷处理更加透明。

### 传统平台的特点

传统平台中，用户主要信任平台本身：

- 资金由平台或支付机构托管。
- 交易状态存储在平台数据库。
- 纠纷处理由平台客服判断。
- 用户很难审计后台规则和处理过程。
- 平台有能力冻结资金、修改状态或调整规则。

### 本项目的优势

1. **资金托管透明**

   买家的钱进入智能合约，而不是直接进入卖家钱包，也不是进入平台内部账户。任何人都可以在链上看到资金是否已托管、是否已放款、是否已退款。

2. **规则公开可查**

   合约中明确规定：

   - 卖家不能提前拿钱。
   - 买家不能单方面退款。
   - 只有买家确认收货后才能正常放款。
   - 退款需要卖家同意。
   - 纠纷需要仲裁多数票。

3. **平台无法挪用资金**

   资金由合约控制，平台方不能绕过合约把钱转走。

4. **纠纷处理可审计**

   仲裁员地址、投票结果、最终资金流向都可以记录在链上，减少传统平台客服黑箱处理的问题。

5. **适合无平台或弱平台场景**

   例如校园二手群、Discord 社群、Telegram 社群、小型兴趣社区、跨国个人交易等。这些场景不一定有成熟平台担保，智能合约可以作为独立担保工具。

### 项目局限

- 智能合约不能自动判断线下商品真假和质量。
- 用户需要使用钱包，存在一定门槛。
- 链上交易需要 gas 费用。
- 仲裁机制需要良好的社区治理。
- 商品展示、聊天、推荐等体验不如成熟中心化平台。

### 关于“去中心化”的诚实定位

需要正面说明，避免被评委追问时立场崩塌：

- 本项目去中心化的是 **资金控制权与规则执行**——平台动不了托管的钱、规则在链上写死、资金流向全程可审计。
- 本项目 **不去中心化“事实判定”**。商品真假链上无法证明，仲裁员看的仍是链下证据，本质是一次主观判断。我们的贡献是把这个判断 **多签化 + 证据哈希锁定 + 全程可追溯**，而不是宣称“无需信任”。
- 仲裁“3 人多签防单点”的前提是 3 个独立的人。合约只能校验“质押过保证金的地址”，理论上一个人可用 3 个地址占满 3 席（女巫攻击）。课堂作业无法在链上彻底解决（缺真实身份），只能靠 **提高质押门槛 + 仲裁员白名单由治理产生** 来缓解，报告中如实说明，不假装漏洞不存在。

把卖点收敛为“**可审计的去中心化托管层**”，而非“无需信任的交易平台”，立论更稳。

## 4. 履约与超时处理（双向）

交易里有两条对称的“拖延”风险，必须各自有超时兜底，并且 **两个计时器串联、不并行**：

- 买家收到货后一直不确认收货 → 卖家钱被锁。
- 买家付款后卖家一直不交付 → 买家钱被锁。

为此引入两个计时窗，并用状态机把它们串起来（见第 8 节的 `Delivered` 状态）。

### 4.1 卖家交付计时（deliveryWindow）

1. 买家付款后，合约记录 `paidAt`，进入 `Locked`。
2. 卖家交付后调用 `markDelivered()`，进入 `Delivered`。
3. 设置交付期限 `deliveryWindow`（实际 7 天，Demo 设 3 到 5 分钟），从 `paidAt` 起算。
4. 若超过 `deliveryWindow` 卖家仍未 `markDelivered`，买家可申请退款或直接升级为纠纷。

这样不接入真实物流，也能在链上体现“卖家也有履约义务”。

### 4.2 买家确认计时（confirmWindow）

1. **`confirmWindow` 只在 `delivered == true`（进入 `Delivered`）之后才开始计时**，与 `deliveryWindow` 串行而非并行。
2. 设置确认期限（实际 7 天，Demo 设 3 到 5 分钟）。
3. 买家在期限内确认收货，合约记账放款给卖家。
4. 买家既不确认也不退款、超期后，卖家可调用 `releaseAfterTimeout()` 超时放款。
5. **超时放款必须校验卖家已 `markDelivered`**（`require(item.delivered)`）。否则会出现“卖家从头到尾没发货，仅靠买家离线就把钱拿走”的攻击——这是把买家保护反向变成卖家武器，必须堵住（见 14.1.1）。
6. 如果买家已经发起退款，则不能超时放款，进入退款或仲裁流程。

### 设计目的

这个机制双向保护：

- 买家付款后，卖家不能立刻拿钱；不交付也拿不到钱。
- 卖家交付后，买家不能无限期拖延确认。
- 自声明交付虽弱，但逼卖家在链上“说谎留证据”，买家可据此发起纠纷。

## 5. 退款和恶意退款处理

智能合约无法直接判断商品是否与描述一致，因此不能允许买家单方面退款。

项目采用两层处理方式：

- 方案 A：双方确认退款。
- 方案 C：DAO / 多签仲裁处理纠纷。

### 双方确认退款

适用于买卖双方没有争议的情况。

流程：

1. 买家付款后发现需要退款。
2. 买家调用 `requestRefund()` 发起退款申请。
3. 卖家同意退款，调用 `approveRefund()`。
4. 合约把托管资金退还给买家。
5. 交易状态变为 `Inactive`。

这种方式可以处理：

- 卖家临时无货。
- 双方协商取消交易。
- 商品确实有问题，卖家愿意退款。

### 恶意退款 / 纠纷处理

如果买家申请退款，但卖家认为买家已经收到商品且属于恶意退款，则交易进入纠纷流程。

流程：

1. 买家申请退款，卖家不同意。
2. 买家或卖家调用 `openDispute(itemId, evidenceHash)` 升级为 `Disputed`，**附带证据哈希并预付固定仲裁费 `arbitrationFee`**（很小，例如 `0.001 ETH`）。
3. `Disputed` 状态下，买卖双方都可调用 `submitEvidence(itemId, evidenceHash)` 各自固定一份证据哈希（链上只存 32 字节哈希，原文留链下，任何人可重算比对，证明证据未被调包）。
4. 仲裁员查看链下证据，调用 `voteDispute` 投票支持买家或卖家。
5. 达到 2/3 多数票后，合约自动执行裁决。

裁决结果：

| 裁决结果 | 合约动作 |
|----------|----------|
| 支持买家 | 退款给买家 |
| 支持卖家 | 放款给卖家 |

### 证据不可篡改

证据原文放链下、仲裁开始后可被替换，则“可审计”站不住。因此发起纠纷与补充举证都把证据内容（或证据打包文件 / IPFS CID）的 `keccak256` 哈希提交上链固定；复核时用同样文件重算哈希与链上比对即可验真。成本极低（一个 `bytes32`），却让“纠纷证据不可篡改、双方平等举证”真正落地。

### 仲裁超时兜底（防资金锁死）

3 人仲裁若只有 1 人投票、或 1:1 后第三人始终不投，交易会永久卡在 `Disputed`、托管资金锁死。因此仲裁也必须有超时：

1. `openDispute()` 记录 `disputeStartedAt`，设置仲裁期限 `disputeWindow`（实际 7 天，Demo 设 3 到 5 分钟）。
2. 期限内达到 2/3 多数票，按多数票执行裁决。
3. 期限内未达多数票（含无人投票），任何人可调用 `resolveDisputeTimeout()` 按 **兜底规则** 结算。

**兜底默认方向结合“卖家是否交付”**（而非无条件偏向买家）：

| 卖家是否 `markDelivered` | 超时兜底结算 | 理由 |
|--------------------------|--------------|------|
| 未交付 | 默认退款买家 | 卖家未履约，保护付款方 |
| 已声明交付 | 默认放款卖家 | 否则恶意买家可“收货→立刻 openDispute→仲裁员怠投→超时→白嫖” |

这样和第 4 节“超时放款须校验交付”天然咬合，堵掉恶意买家必胜路径（见 14.2.1）。

### 仲裁费去向

- 裁决正常完成：仲裁费平均分给 **投了多数票一方** 的仲裁员（投对才有奖励，投少数票 / 没投票不分钱），用“拿不到奖励”形成软约束，无需复杂罚没。
- **超时兜底、无人投票时根本不存在多数票仲裁员**：此时 `arbitrationFee` 原路退还发起方，避免这笔钱锁死在合约里（见 14.1.3）。

兜底结算同样发出事件，状态进入 `Inactive`。

## 6. 仲裁员机制

课程 Demo 中采用固定仲裁委员会，后续可扩展为 DAO 选举仲裁员。

### 仲裁员来源

在校园二手交易场景中，仲裁员可以来自：

- 学生会成员
- 社团负责人
- 二手交易群管理员
- 班委或助教
- 高信誉老用户

课程 Demo 中可以由合约部署者预设 3 个仲裁员地址。

### 投票规则

采用 3 人仲裁委员会：

- 每个仲裁员每笔交易只能投 1 票。
- 投票选项只有两个：支持买家退款，或支持卖家放款。
- 3 个仲裁员中至少 2 个投同一结果，即执行裁决。

### 仲裁员回避

交易当事人不能当本笔交易的仲裁员（校园场景里“群管理员既是卖家又当仲裁员”很现实，会自己给自己投票）。`voteDispute` 里加两行校验即可：

```solidity
require(msg.sender != item.buyer && msg.sender != item.seller, "party cannot vote");
require(isActiveArbiter(msg.sender), "not arbiter");
```

### 有效仲裁员数量

维护 `activeArbiterCount`，质押成功加一、退出减一；`openDispute()` 要求 `activeArbiterCount >= 3`，防止“名义上多签、实际无人可投”。

### 为什么不用单个仲裁员

单个仲裁员容易出现偏袒、误判或被收买。多签仲裁可以降低单点作恶风险。注意其前提是 3 个独立的人，女巫攻击的缓解见第 3 节“诚实定位”。

## 7. 保证金与激励机制

担保要两边对称、仲裁要有动力，因此本项目同时设计了 **仲裁员保证金 + 仲裁费激励 + 卖家保证金 + 举报约束**。

### 7.1 仲裁员保证金

1. 部署者设置候选仲裁员地址。
2. 仲裁员调用 `stakeAsArbiter()` 支付固定保证金，`activeArbiterCount` 加一，地址获得投票资格。
3. 只有已质押的地址才能 `voteDispute`。
4. 仲裁员退出需调用 `withdrawArbiterStake()` 取回保证金。

价值：增加作恶成本、避免随意注册、提高参与责任感、为信誉与治理留扩展空间。

### 7.2 仲裁员退出时机（防纠纷期间逃避）

维护全局 `activeDisputeCount`：`openDispute()` 加一，裁决 / 超时兜底 / 退款完成时减一。**只有 `activeDisputeCount == 0` 时仲裁员才能退出取回保证金**（`require(activeDisputeCount == 0)`），防止仲裁员在纠纷期间抽身、削弱约束。Demo 用全局计数即可，不必为每个仲裁员维护待处理列表。

### 7.3 仲裁费激励闭环

仅质押不奖励，理性仲裁员会选择不投票，反而加剧资金锁死。最简激励：

- **来源**：`openDispute()` 时发起方预付固定 `arbitrationFee`（如 `0.001 ETH`）打入合约。
- **发放**：裁决完成后平分给“投了多数票一方”的仲裁员，投少数票 / 没投票的不分钱。
- **软约束**：保证金本身不动（避免复杂治理），仅靠“投错 / 不投 = 拿不到奖励”约束。
- **败诉方承担**：胜诉方全额拿回托管款，仲裁费实质由败诉方那部分扣回；超时无人投票则原路退发起方（见第 5 节）。

一个 `arbitrationFee` 字段 + 投票时记录 `votedFor`，即可闭环“激励投票 + 惩罚乱投”，无需 DAO。

### 7.4 卖家保证金（让信任对称）

只有买家锁钱、仲裁员质押，卖家零成本就能发布虚假 / 不存在的商品。因此卖家发布或成交时质押一小笔保证金：正常完成后退回；若仲裁判定卖家恶意，则罚没（可并入仲裁费奖励池）。这让担保两边对称。

### 7.5 举报约束与仲裁员反向保护

**核心原则**：举报只是“标记”，不是“判决”。诚实行使职权的仲裁员，无论被谁举报都不该被罚——否则没人敢当仲裁员。

- 严格区分“裁错了”（主观，永不罚没）和“作恶了”（客观可证：长期不投票、可证贿赂串通、违背链上铁证反向投票）。
- 与多数票一致的仲裁员天然安全，集体决策是参照系，不单独追责。
- 举报触发复核，被举报者有权申辩；复核拿不出客观违规证据则维持原状、仲裁员无损。

**举报需质押保证金（反恶意泄愤举报）**，资金部分复用 pull 提款；判定采用 **A 资金机制 + B 自动判定** 组合：

- “不投票”类举报走 **B 全自动**：合约直接查链上事实 `!hasVoted[itemId][arbiter]` 判成立，无需任何人判主观对错。
- “作恶”类举报走 **A 手动**：`resolveReport` 加 `onlyOwner`，由委员会人工判定。
- 举报成立则退还举报人押金（另可从被举报仲裁员保证金扣一部分补偿）；举报不成立则押金罚没，补偿被冤枉的仲裁员。

不引入“复核陪审团”（方案 C），避免“谁监督监督者”的无限递归。

### 7.6 信誉雏形

链上记录买卖双方的完成 / 纠纷次数，校园场景很贴切，也为后续治理与信誉分打基础。

### 后续扩展（不在本次范围）

- 仲裁员信誉分、DAO 投票选举 / 移除仲裁员。
- 对长期不投票的仲裁员真正罚没保证金（本次仅软约束）。
- 复核陪审团式举报判定。
- 稳定币计价（ETH 波动大，USDC 更贴近真实二手交易；本次不做，报告提一句）。

## 8. 推荐合约状态机

```solidity
enum State {
    Created,    // 商品已创建，等待买家付款（可改价 / 下架）
    Locked,     // 买家已付款，资金托管中，等待卖家交付
    Delivered,  // 卖家已声明交付，确认收货倒计时开始
    Disputed,   // 退款纠纷中，等待仲裁
    Inactive    // 交易结束，已放款 / 已退款 / 已取消
}
```

> 加入 `Delivered` 是为了把三个计时窗（`deliveryWindow` / `confirmWindow` / `disputeWindow`）理顺成串行：`confirmWindow` 只在进入 `Delivered` 后才起算，避免“几个超时是并行还是串行”被问倒（见 14.1.2）。

### 状态流转

正常交易：

```text
Created -> Locked -> Delivered -> Inactive   // 放款给卖家
```

下架取消（付款前）：

```text
Created -> Inactive
```

协商退款：

```text
Created -> Locked [-> Delivered] -> Inactive   // 退款给买家
```

纠纷仲裁 / 超时兜底：

```text
Created -> Locked [-> Delivered] -> Disputed -> Inactive
```

超时放款（须卖家已交付）：

```text
Created -> Locked -> Delivered -> Inactive
```

### 合约安全（核心打分项）

所有 ETH 转出（放款、退款、超时放款、仲裁执行）都遵循：

- **Checks-Effects-Interactions**：先改状态（置 `Inactive`）再动资金。
- **`ReentrancyGuard`（`nonReentrant`）** 防重入。
- **pull 提款**：不直接 `transfer`，先记账 `pendingWithdrawals[addr]`，收款方自行 `withdraw()`，避免对方为会 revert 的合约地址时 DoS 卡死交易。

### 基础校验

- `createItem` / `updateItem`：`require(bytes(name).length > 0)`、`require(price > 0)`，描述可空。
- `pay`：`require(state == Created)`、`require(buyer == address(0))`、`require(msg.sender != seller)`、`require(msg.value == price)`，保证每个商品只对应一笔托管、不可重复购买。

### 管理员权限边界（回应中心化后门质疑）

部署者只负责初始化 / 调整候选仲裁员，**不能**：转走买家托管资金、强制改状态为 `Inactive`、替买家确认收货、替卖家同意退款。把“必要的 Demo 管理权限”和“资金控制后门”严格区分。

## 13. 评审视角补充：风险与改进点

> 这一节针对评委最可能质疑的漏洞做补充。前 8 项是必须堵的洞，后 3 项是加分项。对其中“证据上链”和“仲裁激励闭环”给出最简实现方案。

### 13.1 仲裁投票超时兜底（致命洞，必做）

**问题**：3 人仲裁，如果只有 1 人投票、或 1:1 后第三人始终不投，交易会永久卡在 `Disputed`，托管资金锁死。文档原本只给了“确认收货”的超时，**没有给“仲裁”的超时**。

**处理规则**：

1. `openDispute()` 时记录 `disputeStartedAt`。
2. 设置仲裁期限 `disputeWindow`（实际 7 天，Demo 设 3-5 分钟）。
3. 期限内达到 2/3 多数票，按多数票执行裁决。
4. 期限内未达多数票（含无人投票），任何人可调用 `resolveDisputeTimeout()`，按**预设兜底规则**结算：买家已锁定资金且卖家未能赢得多数票，**默认退款给买家**（“疑罪从无”，保护付款方）。
5. 兜底结算同样发出事件，状态进入 `Inactive`。

新增接口：

```solidity
resolveDisputeTimeout(uint256 itemId) // 仲裁超期后任何人可调用，按兜底规则结算
```

### 13.2 仲裁员激励闭环（最简实现）

**问题**：核心版仲裁员只质押保证金、无任何奖励，罚没又被推到“未来扩展”。结果仲裁员只有成本没有收益，理性选择是不投票，反而加剧 13.1 的僵局。

**最简实现（推荐课程项目就做这一版）**：

- **奖励来源**：开启纠纷时，由**发起方预付一笔固定仲裁费** `arbitrationFee`（很小，例如 `0.001 ETH`），随 `openDispute()` 一起打入合约。
- **奖励发放**：裁决完成后，这笔仲裁费**平均分给“投了多数票一方”的仲裁员**（即投对的人才有奖励）。投少数票的、没投票的，不分钱。
- **简单罚没**：保证金本身不动（避免复杂治理），仅靠“投错/不投 = 拿不到奖励”形成软约束。
- **败诉方承担成本**：仲裁费由败诉方实质承担——赢的一方全额拿回托管款，仲裁费从败诉方那部分扣。课程 Demo 可简化为“发起方先垫付，胜诉则从对方应得金额里补回”。

这样用一个 `arbitrationFee` 字段 + 投票时记录 `votedFor`，就能闭环“激励投票 + 惩罚乱投”，无需 DAO 和复杂罚没逻辑。

```solidity
openDispute(uint256 itemId, bytes32 evidenceHash) payable // 附带 arbitrationFee
// 裁决执行时：把 arbitrationFee 按多数票仲裁员人数均分记账，供其 withdraw
```

### 13.2.1 仲裁员的反向保护：被举报怎么办

**问题**：如果诚实仲裁员会因为被举报就受罚，那没人敢当仲裁员——败诉方一定会举报投票反对自己的人。监督机制若设计不好，本身就是寒蝉效应。

**核心原则**：**举报只是“标记”，不是“判决”。诚实行使职权的仲裁员，无论被谁举报，都不该被罚。**

**规则**：

1. **严格区分“裁错了”和“作恶了”**。商品真假本就无法链上证明，所以“你判错了”永远不能作为罚没理由。
   - 可罚没（客观可证）：长期不投票（liveness 链上可验）、可证的贿赂/串通、违背链上铁证反向投票。
   - 不可罚没（主观）：投了少数票、判决让一方不满、对证据判断不同。
2. **多数票护盾**：跟多数票一致的仲裁员天然安全，集体决策是参照系，不单独追责。
3. **举报 ≠ 自动罚没**：举报只触发复核流程，由独立方（Demo 中为部署者/委员会）判定，被举报者有权提交申辩证据。
4. **默认无罪**：复核拿不出客观违规证据时，维持原状、仲裁员无损。

Demo 中最简实现：`reportArbiter()` 只发事件、记录在链上，不直接动保证金；罚没权限收口到治理，且只对“不投票/可证作恶”罚没。诚实但被举报的仲裁员，结果就是“留个链上记录，毫发无损”。

### 13.2.2 举报保证金与判定（反恶意举报）

**目的**：让败诉方不能零成本地泄愤式举报。举报需质押一笔保证金，举报不成立则罚没（补偿被冤枉的仲裁员）。

**最终方案：A 资金机制 + B 自动判定。** “不投票”类举报走 B 全自动（合约查链上事实），“作恶”类举报走 A 由委员会手动 `resolveReport`。既有“举报要付费、乱举报会被罚”的完整故事，又不掉进“谁监督监督者”的复核陪审团递归坑。复杂度低到中。

**资金机制（A）**：复用 13.4 的 pull 提款。

```solidity
struct Report {
    address reporter;
    address arbiter;
    uint256 deposit;
    bytes32 evidenceHash;
    bool    resolved;
}

function reportArbiter(uint256 itemId, address arbiter, bytes32 evidenceHash)
    external payable
{
    require(msg.value == reportDeposit, "wrong deposit"); // 固定举报保证金
    reports[itemId] = Report(msg.sender, arbiter, msg.value, evidenceHash, false);
    emit ArbiterReported(itemId, msg.sender, arbiter, evidenceHash);
}

function resolveReport(uint256 itemId, bool founded) external {
    Report storage r = reports[itemId];
    require(!r.resolved); r.resolved = true;
    if (founded) {
        pendingWithdrawals[r.reporter] += r.deposit;   // 退还押金（另可从仲裁员保证金扣一部分补偿举报人）
    } else {
        pendingWithdrawals[r.arbiter]  += r.deposit;   // 押金罚没，补偿被冤枉的仲裁员
    }
}
```

**判定机制**：

- **A 手动判定**（“作恶”类）：`resolveReport` 加 `onlyOwner`，成不成立由委员会人工填，被举报者可申辩。
- **B 自动判定**（“不投票”类）：只认“客观可证”事实，合约自查链上 `hasVoted`，无需任何人判主观对错。

方案 B 示例（“不投票”自动成立）：

```solidity
function resolveReport(uint256 itemId) external {
    Report storage r = reports[itemId];
    require(disputes[itemId].resolved, "dispute not resolved");
    bool founded = !hasVoted[itemId][r.arbiter];  // 链上事实
    ...
}
```

### 13.3 仲裁员回避机制（必做，改动极小）

**问题**：没有禁止交易当事人作为该笔交易的仲裁员。校园场景里“群管理员既是卖家又当仲裁员”很现实，会导致自己给自己投票。

**实现**：`voteDispute` 里加一行校验即可。

```solidity
require(msg.sender != item.buyer && msg.sender != item.seller, "party cannot vote");
require(isActiveArbiter(msg.sender), "not arbiter");
```

### 13.4 合约安全：重入保护 + 转账模式（必做，核心打分项）

**问题**：放款、退款、超时放款、仲裁执行都涉及 ETH 转出，文档完全没提安全。

**要求**：

- 遵循 **Checks-Effects-Interactions**：先改状态（置 `Inactive`）再转账。
- 加 **`ReentrancyGuard`**（OpenZeppelin `nonReentrant`）。
- **放款用 pull 提款模式**：不直接 `transfer` 给对方，而是先记账到 `pendingWithdrawals[addr]`，由收款方自己调用 `withdraw()` 取走。否则若卖家是会 revert 的合约地址，可 DoS 卡死整笔交易。

```solidity
mapping(address => uint256) public pendingWithdrawals;
withdraw() // 收款方主动提走自己应得的资金
```

### 13.5 卖家保证金（可选，让信任对称）

**问题**：现在只有买家锁钱、仲裁员质押，**卖家零成本**，可随意发布虚假/不存在商品，最坏只是交易作废。

**实现**：卖家发布或成交时质押一小笔保证金，正常完成后退回；若仲裁判定卖家恶意，则罚没（可并入仲裁费奖励池）。这让担保的两边对称，是很好的讲故事点。

### 13.6 取消 / 下架接口（必做，逻辑简单）

**问题**：没有 `cancelItem`。买家付款前卖家想下架/改价怎么办？买家点了购买但卡住想撤销怎么办？

**实现**：仅在 `Created` 状态允许卖家下架；`Locked` 后不可单方取消（走退款/仲裁）。

```solidity
cancelItem(uint256 itemId) // 仅 Created 状态、仅卖家可调用
```

### 13.7 证据 hash 上链固定（最简实现，把“可审计”坐实）

**问题**：证据放链下，仲裁开始后**可被替换**，链上无法证明仲裁时看的是哪份证据，“可审计”这个卖点站不住。

**最简实现**：不需要真上传 IPFS，只需在发起纠纷时把证据内容（或证据打包文件 / IPFS CID）的 **keccak256 哈希提交上链固定**：

```solidity
openDispute(uint256 itemId, bytes32 evidenceHash) // evidenceHash = keccak256(证据文件)
```

- 链下保存原始证据文件；链上只存 32 字节哈希。
- 仲裁/复核时，任何人用同样的文件重新计算哈希，和链上比对即可验证“证据未被调包”。
- 成本极低（一个 `bytes32`），却让“纠纷证据不可篡改”真正落地。
- 双方都想提交证据时，可在 `voteDispute` 前各自调用一次 `submitEvidence(itemId, evidenceHash)` 分别记录买卖双方哈希。

### 13.8 用 Event 区分交易结局（必做）

**问题**：状态机里“正常放款”和“退款成功”都是 `Created -> Locked -> Inactive`，只看 `State` 无法区分交易最终结局。

**实现**：靠事件区分，前端时间线和链上审计都依赖它。

```solidity
event Released(uint256 indexed itemId, address indexed seller, uint256 amount); // 放款给卖家
event Refunded(uint256 indexed itemId, address indexed buyer, uint256 amount);  // 退款给买家
event DisputeResolved(uint256 indexed itemId, bool supportBuyer, bool byTimeout);
```

### 13.9 加分项（有时间再做）

- **信誉系统雏形**：链上记录买卖双方的完成 / 纠纷次数，校园场景很贴切。
- **稳定币计价**：ETH 波动大，支持 USDC 更贴近真实二手交易（Demo 可不做，报告提一句）。

### 13.11 评委可能继续追问的低成本漏洞与解决方案

以下问题不一定都是核心主流程，但都属于评委容易追问、实现成本不高、适合课堂报告补充的边界点。

#### 13.11.1 商品信息修改和冻结

**问题**：卖家创建商品后，如果还能随意改价格或描述，买家付款前后看到的信息可能不一致。评委可能会问：卖家能不能在买家付款后偷偷改价或改描述？

**解决方案**：

1. `Created` 状态下允许卖家修改商品名称、描述和价格。
2. 一旦买家付款进入 `Locked` 状态，商品信息冻结，不允许继续修改。
3. 每次修改发出 `ItemUpdated` 事件，方便前端和报告展示修改记录。

推荐接口：

```solidity
updateItem(uint256 itemId, string name, string description, uint256 price)
```

关键校验：

```solidity
require(msg.sender == item.seller, "only seller");
require(item.state == State.Created, "item locked");
require(price > 0, "invalid price");
```

#### 13.11.2 卖家不交付商品

**问题**：文档已经处理了“买家不确认收货”，但如果买家付款后卖家一直不交付商品，也会造成买家资金被动锁定。

**解决方案**：

1. 买家付款后记录 `paidAt`。
2. 卖家可以调用 `markDelivered()` 或 `markShipped()` 标记已交付。
3. 设置交付期限 `deliveryWindow`，Demo 中可设置为 3 到 5 分钟。
4. 如果超过期限卖家仍未标记交付，买家可以申请退款或直接升级为纠纷。

推荐接口：

```solidity
markDelivered(uint256 itemId)
```

这样不需要接入真实物流，也能在状态机中体现“卖家也有履约义务”。

#### 13.11.3 有效仲裁员数量不足

**问题**：方案默认 3 人仲裁、2/3 多数通过。但如果只有 1 个或 2 个仲裁员完成保证金质押，纠纷机制就不是完整的多签仲裁。

**解决方案**：

1. 维护 `activeArbiterCount`。
2. `stakeAsArbiter()` 成功后增加有效仲裁员数量。
3. `withdrawArbiterStake()` 成功后减少有效仲裁员数量。
4. `openDispute()` 时要求有效仲裁员数量至少为 3。

关键校验：

```solidity
require(activeArbiterCount >= 3, "not enough arbiters");
```

这可以防止“名义上多签，实际上无人可投”的情况。

#### 13.11.4 仲裁员退出保证金时机

**问题**：如果仲裁员在纠纷期间取回保证金并退出，就会削弱保证金约束，也可能导致纠纷无法完成。

**解决方案**：

1. 维护全局 `activeDisputeCount`。
2. `openDispute()` 时加 1。
3. 纠纷裁决、超时兜底或退款完成时减 1。
4. 只有 `activeDisputeCount == 0` 时，仲裁员才允许退出并取回保证金。

关键校验：

```solidity
require(activeDisputeCount == 0, "dispute pending");
```

课程 Demo 中用全局计数即可，不必为每个仲裁员维护复杂的待处理列表。

#### 13.11.5 重复购买和商品已售出

**问题**：同一个商品如果已经被买家付款，第二个买家不应该还能继续购买。这个问题属于状态机基础漏洞，评委很容易检查。

**解决方案**：

`pay()` 中必须同时校验状态和买家地址：

```solidity
require(item.state == State.Created, "not available");
require(item.buyer == address(0), "already has buyer");
require(msg.sender != item.seller, "seller cannot buy");
require(msg.value == item.price, "wrong price");
```

这样可以保证每个商品只对应一笔托管交易。

#### 13.11.6 输入校验：空商品和零价格

**问题**：卖家创建空名称、空描述或价格为 0 的商品，会让 Demo 显得不严谨，也可能干扰前端展示和测试。

**解决方案**：

`createItem()` 和 `updateItem()` 都应加入基础输入校验：

```solidity
require(bytes(name).length > 0, "empty name");
require(price > 0, "invalid price");
```

描述可以允许为空，也可以限制最大长度。课堂 Demo 中建议保留描述字段但不做复杂长度校验，避免合约过度复杂。

#### 13.11.7 事件日志覆盖完整流程

**问题**：如果事件不完整，前端事件流和报告中的“可追溯”卖点会变弱。只靠 `State` 也无法解释每一步发生了什么。

**解决方案**：

为所有关键动作补充事件：

```solidity
event ItemCreated(uint256 indexed itemId, address indexed seller, uint256 price);
event ItemUpdated(uint256 indexed itemId);
event ItemCancelled(uint256 indexed itemId);
event ItemPaid(uint256 indexed itemId, address indexed buyer, uint256 amount);
event RefundRequested(uint256 indexed itemId, address indexed buyer);
event DisputeOpened(uint256 indexed itemId, bytes32 evidenceHash);
event EvidenceSubmitted(uint256 indexed itemId, address indexed submitter, bytes32 evidenceHash);
event ArbiterVoted(uint256 indexed itemId, address indexed arbiter, bool supportBuyer);
event StakeDeposited(address indexed arbiter, uint256 amount);
event StakeWithdrawn(address indexed arbiter, uint256 amount);
```

前端可以直接监听这些事件生成操作日志，课堂演示时也更直观。

#### 13.11.8 买卖双方都能提交证据

**问题**：如果只有发起纠纷的一方能提交证据 hash，另一方会缺少链上固定证据的机会，不利于体现公平仲裁。

**解决方案**：

1. `Disputed` 状态下，买家和卖家都可以提交证据 hash。
2. 证据原文仍保存在链下，链上只保存 `bytes32` hash。
3. 每次提交发出 `EvidenceSubmitted` 事件。

推荐接口：

```solidity
submitEvidence(uint256 itemId, bytes32 evidenceHash)
```

关键校验：

```solidity
require(item.state == State.Disputed, "not disputed");
require(msg.sender == item.buyer || msg.sender == item.seller, "not party");
require(evidenceHash != bytes32(0), "empty evidence");
```

这样可以用很低成本补强“证据不可篡改”和“双方平等举证”。

#### 13.11.9 管理员权限边界

**问题**：如果部署者可以设置仲裁员，评委可能会担心项目仍然有中心化后门：管理员是否能转走托管资金、强制改状态或替用户确认收货？

**解决方案**：

报告和合约中都应明确：

1. 部署者只负责初始化或调整候选仲裁员。
2. 部署者不能转走买家的托管资金。
3. 部署者不能强制把交易状态改成 `Inactive`。
4. 部署者不能替买家确认收货。
5. 部署者不能替卖家同意退款。

这样可以把“必要的 Demo 管理权限”和“资金控制后门”区分开。

#### 13.11.10 前端错误提示和演示稳定性

**问题**：课堂演示中，MetaMask 网络、账户、余额或用户拒签都可能导致操作失败。如果前端没有提示，评委会觉得系统不可用。

**解决方案**：

前端至少处理以下场景：

1. 未连接钱包。
2. 当前网络不是 Sepolia。
3. 当前账户不是该操作所需的卖家、买家或仲裁员。
4. 余额不足或付款金额错误。
5. 用户在 MetaMask 中拒绝交易。
6. 交易 pending、success、failed 三种状态。

这些属于前端交互层的轻量补强，不改变合约核心逻辑，但能显著提高 Demo 稳定性。

### 13.12 第二轮评审补充优先级（已定稿，全做）

| 改进点 | 价值 | 最终决定 |
|--------|------|----------|
| 重复购买校验、零价格校验、商品锁定后不可修改 | 状态机和输入校验基础安全 | ✅ 做 |
| 管理员不能动托管资金、不能替用户结算 | 回应中心化后门质疑 | ✅ 做 |
| 卖家交付标记和交付超时处理 | 补齐“卖家不交付”场景 | ✅ 做 |
| 有效仲裁员数量不足时不能开启纠纷 | 让 3 人仲裁机制真实成立 | ✅ 做 |
| 仲裁员有未完成纠纷时不能退出保证金 | 防止纠纷期间逃避约束 | ✅ 做 |

| 前端完整错误提示 | 提升课堂演示稳定性 | ✅ 做 |

## 14. 第三轮评审：仍未闭合的致命洞与必答定位题

> 前面 13 节已经把常见漏洞堵得很全。这一节只收录**目前文档仍未真正闭合、评委大概率会现场追问**的点，并标注课堂作业的实现难度。分三类：致命且好实现、致命但需设计权衡、免费但必答的定位题。

### 14.1 致命 + 好实现（优先补）

#### 14.1.1 超时放款不校验卖家是否交付（最该补）

**问题**：13.11.2 加了 `markDelivered()`，但**没有和超时放款 `releaseAfterTimeout()` 串起来**。当前逻辑是：买家付款 → `confirmWindow` 到期 → 卖家直接调 `releaseAfterTimeout()` 拿钱。结果是**卖家从头到尾没发货**，只要买家忙/离线、没在窗口内 `openDispute`，卖家照样把钱拿走。这把"买家不确认"的保护反过来变成了卖家的攻击手段，是目前最大的逻辑漏洞，一个反例就能在演示现场被打穿。

**解决方案（一行 require，顺便让 `markDelivered` 真正有用）**：

```solidity
function releaseAfterTimeout(uint256 itemId) external {
    require(item.delivered, "not delivered");                 // 卖家必须先自声明交付
    require(block.timestamp >= paidAt + confirmWindow, "too early");
    ...
}
```

自声明交付虽弱，但至少逼卖家在链上"说谎留证据"，买家可凭此发起纠纷。**难度：极低，价值极高。**

#### 14.1.2 多个计时器没串联 + 状态机缺 Delivered 状态

**问题**：补充章节陆续引入了三个计时窗——`confirmWindow`（确认）、`deliveryWindow`（交付，13.11.2）、`disputeWindow`（仲裁），但：

- 状态机（第 8 节）只有 `Created / Locked / Disputed / Inactive`，**没有 Delivered 状态**；
- `deliveryWindow` 和 `confirmWindow` 都从 `paidAt` 起算，**同时在跑**，谁优先没有定义。

评委一句"你这几个超时是并行还是串行"就会问倒。

**解决方案**：二选一——

1. 加一个 `Delivered` 状态把流程理顺：`Locked -> Delivered -> Inactive`；
2. 或者不加状态，但明确写死"`confirmWindow` 只在 `delivered == true` 之后才开始计时"。

**难度：低（改时间起算点）到中（加状态）。**

#### 14.1.3 仲裁费在"超时兜底、无人投票"时去向未定义

**问题**：13.1 说仲裁超时无人投票则默认退款买家；13.2 说仲裁费 `arbitrationFee` "平分给投多数票的仲裁员"。两者冲突：超时且无人投票时**根本不存在多数票仲裁员**，这笔仲裁费会留在合约里没人能取，资金锁死——而这正是 13.4 想避免的情况。

**解决方案**：明确"超时兜底结算时，`arbitrationFee` 原路退还发起方"，一行记账即可。**难度：极低。**

### 14.2 致命但需设计权衡（评委爱追问的逻辑洞）

#### 14.2.1 超时兜底"默认退款买家"可被操纵，恶意买家稳赢

**问题**：13.1 的"疑罪从无、默认退款买家"和 13.2 的"仲裁员不投票没收益"叠在一起，产生一条恶意买家必胜路径：

> 买家线下收货 → 立刻 `openDispute` → 仲裁员理性怠投（投了也未必拿到钱）→ 超时 → 默认退款买家 → 买家白嫖。

这恰好放大了 13.1 自己承认的僵局问题。

**解决方案**：超时兜底的默认方向不要无条件偏向买家，而是**结合卖家是否 `markDelivered`**——卖家已声明交付则默认放款卖家，卖家未交付则默认退款买家。这样和 14.1.1 的修复天然咬合。代码不难，但要在报告里讲清为什么这样选。**难度：设计判断为主，代码低。**

#### 14.2.2 女巫攻击：一人多地址霸占 3 个仲裁席

**问题**："3 人多签防单点作恶"的整个卖点，前提是 3 个仲裁员是 3 个独立的人。但合约只校验"质押过保证金的地址"，**一个人完全可以用 3 个地址质押、占满 3 席**，自己给自己凑多数票，多签形同虚设。

**解决方案**：课堂作业无法在链上彻底解决（没有真实身份），但**必须在报告里主动说明**——靠"提高质押门槛 + 仲裁员白名单由治理产生"缓解，而不是假装漏洞不存在。评委最反感"假装多签、实则单点"。**难度：报告说明为主，代码无改动。**

### 14.3 免费但必答的定位题

#### 14.3.1 仲裁本质仍是主观信任转移，而非去中心化

**问题**：最锋利的追问——

> "商品真假你自己说链上证明不了，那仲裁员投票靠的还是主观看链下证据。这和中心化平台客服拍脑袋判，本质区别在哪？你只是把信任从'1 个平台'换成了'3 个你预设的人'。"

文档把它列进了"局限"，但没有正面回应。

**解决方案（纯定位，免费）**：在报告里把话说透——本项目去中心化的是**资金控制权和规则执行**（平台动不了钱、规则链上写死、流向全程可审计），**不是**去中心化"事实判定"。事实判定永远需要人，本项目的贡献是让这个判定**多签化 + 证据哈希锁定 + 全程可追溯**。把卖点收敛到"可审计的托管层"，而非"无需信任"，立论会稳得多。

### 14.4 第三轮评审优先级总结（已定稿，全做）

| 改进点 | 难度 | 最终决定 |
|--------|------|----------|
| 超时放款必须校验卖家已交付（14.1.1） | 极低，一行 | ✅ 做 |
| 仲裁费在超时无人投票时原路退发起方（14.1.3） | 极低，一行 | ✅ 做 |
| 多计时器串联 / 补 Delivered 状态（14.1.2） | 低~中 | ✅ 做 |
| 超时兜底默认方向结合是否交付（14.2.1） | 设计权衡 | ✅ 做 |
| 女巫攻击使多签失效，报告说明缓解（14.2.2） | 报告说明 | ✅ 做 |
| 仲裁=主观信任转移，定位正面回应（14.3.1） | 免费 | ✅ 做 |

## 15. 评委视角：Web3 与区块链风味评估

> 记录日期：2026-06-23
> 目的：为后续项目报告和课堂展示 PPT 固定项目定位，避免把项目讲成“链上版闲鱼”而被追问商品展示、聊天、物流和真假识别。

### 15.1 总体判断

本项目对于评委来说 **Web3 风味足够，区块链风味更扎实**。

项目不应被包装成完整的去中心化二手交易平台，而应定位为：

> 一个面向校园二手交易和小型社区交易的链上担保托管层，用智能合约替代中心化平台的资金托管和结算规则，并通过多签仲裁处理交易纠纷。

更适合在报告和 PPT 中强调：

- 本项目不是重做闲鱼、转转的商品展示和聊天系统。
- 本项目解决的是交易中最关键的资金信任问题。
- 链上部分负责托管资金、执行状态机、记录事件、固定证据哈希、处理放款 / 退款 / 仲裁。
- 链下部分负责图片、聊天、线下交付和证据原文。

### 15.2 Web3 风味评估

本项目的 Web3 风味主要来自：

1. 用户通过 MetaMask 连接钱包并发起交易。
2. 买家、卖家、仲裁员都以链上地址作为身份参与流程。
3. 托管资金由用户和智能合约共同驱动，不依赖平台后台余额。
4. 仲裁员质押保证金后才获得投票资格。
5. 后续可以扩展 DAO 选举仲裁员、链上信誉和社区治理。

评估结论：

```text
Web3 风味：8 / 10
```

如果项目只做到“买家付款进合约、确认后放款”，Web3 感会比较普通；但当前方案加入了多签仲裁、保证金、证据哈希、事件审计、pull 提款、管理员权限边界和超时兜底，因此已经明显超过普通课程 Demo。

### 15.3 区块链风味评估

本项目的区块链风味比 Web3 风味更强，主要来自：

1. **资金托管必须用链**

   买家的 ETH 进入智能合约，而不是进入平台账户或卖家钱包。平台管理员无法绕过合约挪用托管资金。

2. **业务规则被编码成链上状态机**

   `Created -> Locked -> Delivered -> Disputed -> Inactive` 清楚体现智能合约项目的状态流转。

3. **事件日志可追溯**

   创建、付款、交付、退款申请、证据提交、仲裁投票、放款、退款都通过 event 记录，前端可生成完整操作日志。

4. **自动执行结算**

   买家确认收货、卖家超时放款、仲裁 2/3 多数裁决后，合约自动改变资金归属。

5. **多签仲裁和保证金机制**

   仲裁员是链上地址，投票结果公开，质押保证金提高参与门槛，仲裁费激励投票。

6. **证据哈希上链存证**

   原始证据不上链，但证据哈希上链固定，证明某个时间点提交过某份证据，后续可用原文重算哈希验证。

评估结论：

```text
区块链风味：8.5 / 10
```

### 15.4 推荐答辩表达

报告和 PPT 中应避免使用：

```text
我们做了一个去中心化闲鱼。
```

更推荐使用：

```text
我们做的是一个基于 Solidity 的链上担保托管合约，用智能合约替代中心化平台的资金托管和结算规则，并通过多签仲裁处理交易纠纷。
```

或者：

```text
Trust the contract, not the platform.
平台不能动钱，只能展示状态。
仲裁不是消灭信任，而是把单点信任变成可审计的多方约束。
```

### 15.5 展示重点

课堂展示时，应优先展示这些“区块链味”最强的点：

- MetaMask 签名发起交易。
- 买家付款后 ETH 进入合约托管。
- 状态从 `Created` 变为 `Locked`，付款后商品信息冻结。
- 卖家 `markDelivered()` 后进入 `Delivered`，确认收货倒计时开始。
- 买家确认收货后，合约记账给卖家，卖家通过 `withdraw()` 提款。
- 买家不能单方面退款，纠纷需要仲裁员投票。
- 2/3 仲裁员投票后自动执行裁决。
- 证据哈希、投票和最终资金流向都能通过链上事件追溯。

### 15.6 需要主动说明的边界

评委可能会问“商品真假链上无法判断，那和中心化平台有什么区别”。答辩时应主动说明：

- 本项目去中心化的是 **资金控制权与规则执行**，不是商品事实判定。
- 商品真假、质量、线下交付仍需要人和链下证据判断。
- 本项目的贡献是把主观判断变成 **多签化 + 证据哈希锁定 + 全程可追溯**。
- 因此项目定位是“可审计的链上担保托管层”，不是“无需信任的完整二手平台”。
