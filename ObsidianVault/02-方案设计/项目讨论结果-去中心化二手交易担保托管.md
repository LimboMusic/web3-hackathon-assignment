# 项目讨论结果：去中心化二手交易担保托管平台

> 讨论日期：2026-06-23
> 项目方向：Decentralized Escrow System
> 推荐场景：校园二手交易担保平台

---

## 1. 项目定位

本项目不是要完整复刻闲鱼、转转这类二手交易平台，而是做一个面向二手交易的 **链上担保支付层**。

传统二手平台主要解决商品展示、聊天、推荐、物流和客服问题；本项目重点解决的是交易中最关键的 **资金信任问题**：

- 买家怕先付款后卖家不发货。
- 卖家怕先发货后买家不付款。
- 双方都不希望完全依赖中心化平台后台。
- 纠纷处理希望更加公开、可追溯。

项目核心价值：

> 买家的资金先进入智能合约托管，卖家不能提前拿走；买家确认收货后，合约自动放款。如果发生退款纠纷，则通过多签仲裁和保证金机制处理。

## 2. 推荐实现场景

### 场景：校园二手交易

学生 A 想出售一台二手显示器，学生 B 想购买。双方可以通过微信群、校园论坛或线下方式沟通，但资金流转交给智能合约处理。

### 正常交易流程

1. 卖家发布商品，例如“二手显示器”，价格为 `0.01 ETH`。
2. 买家连接 MetaMask，点击购买。
3. 买家支付的 ETH 进入智能合约托管。
4. 合约状态从 `Created` 变为 `Locked`。
5. 卖家线下交付商品。
6. 买家收到商品后点击确认收货。
7. 合约自动把托管资金转给卖家。
8. 交易状态变为 `Inactive`。

### 链上和链下分工

链下负责：

- 商品图片
- 商品描述
- 买卖双方沟通
- 线下交付
- 纠纷证据，例如照片、聊天记录、物流信息

链上负责：

- 卖家地址
- 买家地址
- 商品价格
- 交易状态
- 托管资金
- 放款逻辑
- 退款逻辑
- 仲裁投票
- 保证金记录
- 事件日志

## 3. 和传统平台相比的优势

本项目不强调商品展示和聊天体验超过传统平台，而是强调资金托管和纠纷处理更加透明。

### 传统平台的特点

传统平台中，用户主要信任平台本身：

- 资金由平台或支付机构托管。
- 交易状态存储在平台数据库。
- 纠纷处理由平台客服判断。
- 用户很难审计后台规则和处理过程。
- 平台有能力冻结资金、修改状态或调整规则。

### 本项目的优势

1. **资金托管透明**

   买家的钱进入智能合约，而不是直接进入卖家钱包，也不是进入平台内部账户。任何人都可以在链上看到资金是否已托管、是否已放款、是否已退款。

2. **规则公开可查**

   合约中明确规定：

   - 卖家不能提前拿钱。
   - 买家不能单方面退款。
   - 只有买家确认收货后才能正常放款。
   - 退款需要卖家同意。
   - 纠纷需要仲裁多数票。

3. **平台无法挪用资金**

   资金由合约控制，平台方不能绕过合约把钱转走。

4. **纠纷处理可审计**

   仲裁员地址、投票结果、最终资金流向都可以记录在链上，减少传统平台客服黑箱处理的问题。

5. **适合无平台或弱平台场景**

   例如校园二手群、Discord 社群、Telegram 社群、小型兴趣社区、跨国个人交易等。这些场景不一定有成熟平台担保，智能合约可以作为独立担保工具。

### 项目局限

- 智能合约不能自动判断线下商品真假和质量。
- 用户需要使用钱包，存在一定门槛。
- 链上交易需要 gas 费用。
- 仲裁机制需要良好的社区治理。
- 商品展示、聊天、推荐等体验不如成熟中心化平台。

## 4. 买家不确认收货的处理

如果买家收到货后一直不点击确认收货，会导致卖家已经交付商品，但资金仍锁在合约中。

为避免买家恶意拖延，系统加入 **自动确认期限**。

### 处理规则

1. 买家付款后，合约记录付款时间 `paidAt`。
2. 交易进入 `Locked` 状态。
3. 设置确认期限，例如实际业务中为 7 天，课堂 Demo 中可设置为 3 到 5 分钟。
4. 如果买家在期限内确认收货，合约正常放款给卖家。
5. 如果买家没有确认收货，也没有发起退款，超过期限后卖家可以调用超时放款。
6. 如果买家已经发起退款，则不能自动超时放款，需要进入退款或仲裁流程。

### 设计目的

这个机制同时保护双方：

- 买家付款后，卖家不能立刻拿钱。
- 卖家交付后，买家不能无限期拖延确认。

## 5. 退款和恶意退款处理

智能合约无法直接判断商品是否与描述一致，因此不能允许买家单方面退款。

项目采用两层处理方式：

- 方案 A：双方确认退款。
- 方案 C：DAO / 多签仲裁处理纠纷。

### 双方确认退款

适用于买卖双方没有争议的情况。

流程：

1. 买家付款后发现需要退款。
2. 买家调用 `requestRefund()` 发起退款申请。
3. 卖家同意退款，调用 `approveRefund()`。
4. 合约把托管资金退还给买家。
5. 交易状态变为 `Inactive`。

这种方式可以处理：

- 卖家临时无货。
- 双方协商取消交易。
- 商品确实有问题，卖家愿意退款。

### 恶意退款 / 纠纷处理

如果买家申请退款，但卖家认为买家已经收到商品且属于恶意退款，则交易进入纠纷流程。

流程：

1. 买家申请退款。
2. 卖家不同意退款。
3. 买家或卖家将交易升级为 `Disputed` 状态。
4. 仲裁员查看双方提交的证据。
5. 仲裁员投票支持买家或卖家。
6. 达到多数票后，合约自动执行裁决。

裁决结果：

| 裁决结果 | 合约动作 |
|----------|----------|
| 支持买家 | 退款给买家 |
| 支持卖家 | 放款给卖家 |

## 6. 仲裁员机制

课程 Demo 中采用固定仲裁委员会，后续可扩展为 DAO 选举仲裁员。

### 仲裁员来源

在校园二手交易场景中，仲裁员可以来自：

- 学生会成员
- 社团负责人
- 二手交易群管理员
- 班委或助教
- 高信誉老用户

课程 Demo 中可以由合约部署者预设 3 个仲裁员地址。

### 投票规则

采用 3 人仲裁委员会：

- 每个仲裁员每笔交易只能投 1 票。
- 投票选项只有两个：支持买家退款，或支持卖家放款。
- 3 个仲裁员中至少 2 个投同一结果，即执行裁决。

### 为什么不用单个仲裁员

单个仲裁员容易出现偏袒、误判或被收买。多签仲裁可以降低单点作恶风险。

## 7. 保证金机制

为了约束仲裁员行为，可以加入保证金机制。

### 基本思路

仲裁员在成为有效仲裁员前，需要向合约质押一定数量的 ETH 作为保证金。

如果仲裁员长期不参与投票、恶意投票或被社区认定为不可信，后续可以通过治理机制扣除部分保证金或取消仲裁资格。

### 在课程项目中的实现边界

为了控制复杂度，课程 Demo 中可以实现轻量版保证金机制：

1. 部署者设置候选仲裁员地址。
2. 仲裁员调用 `stakeAsArbiter()` 并支付固定保证金。
3. 只有已质押保证金的地址才有投票资格。
4. 仲裁员可以在没有待处理纠纷时退出并取回保证金。
5. 如果仲裁员参与恶意行为，项目报告中说明后续可通过 DAO 治理进行罚没。

### 保证金机制的价值

- 增加仲裁员作恶成本。
- 避免随意注册仲裁员。
- 提高仲裁员参与投票的责任感。
- 为后续 DAO 治理和信誉系统留下扩展空间。

### 后续扩展

后续可以增加：

- 仲裁员信誉分。
- DAO 投票选举和移除仲裁员。
- 对长期不投票的仲裁员扣除部分保证金。
- 对被多数社区成员认定为恶意裁决的仲裁员罚没保证金。
- 仲裁员投票奖励机制。

## 8. 推荐合约状态机

```solidity
enum State {
    Created,    // 商品已创建，等待买家付款
    Locked,     // 买家已付款，资金托管中
    Disputed,   // 退款纠纷中，等待仲裁
    Inactive    // 交易结束，已放款或已退款
}
```

### 状态流转

正常交易：

```text
Created -> Locked -> Inactive
```

退款成功：

```text
Created -> Locked -> Inactive
```

纠纷仲裁：

```text
Created -> Locked -> Disputed -> Inactive
```

超时放款：

```text
Created -> Locked -> Inactive
```

## 11. 课堂演示脚本

### 演示一：正常交易

1. 卖家账户发布“二手显示器”，价格 `0.01 ETH`。
2. 买家账户连接钱包并支付。
3. 页面显示资金已进入合约托管，状态为 `Locked`。
4. 买家模拟收到货，点击确认收货。
5. 合约自动放款给卖家，状态变为 `Inactive`。

### 演示二：买家恶意退款

1. 卖家发布“机械键盘”。
2. 买家付款，资金进入合约。
3. 买家申请退款。
4. 卖家不同意，交易进入纠纷状态。
5. 3 个仲裁员中 2 个投票支持卖家。
6. 合约自动把资金释放给卖家。

### 演示三：合理退款

1. 买家付款后，卖家发现商品损坏或无法交付。
2. 买家申请退款。
3. 卖家同意退款。
4. 合约把资金退还给买家。

## 12. 最终建议

本项目建议采用以下实现范围：

- 必做：
  - 商品创建
  - 买家付款托管
  - 买家确认收货放款
  - 买家申请退款
  - 卖家同意退款
  - 超时放款
  - 3 人多签仲裁
  - 仲裁员保证金质押

- 可选：
  - IPFS 证据链接
  - 仲裁员信誉分
  - DAO 投票更换仲裁员
  - 仲裁员投票奖励
  - 保证金罚没

课程项目中，建议先实现“轻量可跑通版本”，把完整 Demo 做稳定；更复杂的 DAO 治理和罚没机制可以放入报告的未来扩展部分。
