# 前端展示流程

> 版本：2026-06-28 初稿  
> 目标：用于课堂 Demo 讲解前端如何展示“不同钱包地址对应不同角色权限”。  
> 展示原则：演示控制台只用于课堂切换账号和场景；真正的业务按钮必须由当前钱包地址推导权限。

---

## 1. 演示角色与账号

课堂演示使用 6 个 Demo 账号模拟 MetaMask 多账号切换：

| 账号 | 页面显示角色 | 用途 |
|------|--------------|------|
| 卖家 Seller | 商品发布者 | 创建商品、标记交付、同意退款、超时放款、响应纠纷 |
| 买家 Buyer | 商品购买者 | 付款托管、确认收货、申请退款、发起纠纷 |
| 仲裁员 A | Arbiter A | 质押后参与投票 |
| 仲裁员 B | Arbiter B | 质押后参与投票 |
| 仲裁员 C | Arbiter C | 展示第三名仲裁员在 2/3 达成后不能继续投票 |
| 访客 Viewer | 只读用户 | 查看交易状态、资金流和事件日志，不能执行业务操作 |

顶部状态栏需要始终展示：

1. 当前 Demo 账号名称。
2. 当前短地址。
3. 推导出的当前角色。
4. 当前网络：Sepolia / Demo Network。
5. 当前交易状态：Idle / Pending / Success / Failed。

---

## 2. 演示控制台

演示控制台放在顶部状态栏下方或页面右上角，名称必须明确为：

```text
课堂演示控制台
```

控制台只做两类事：

1. **切换 Demo 账号**
   - 按钮：`卖家`、`买家`、`仲裁员 A`、`仲裁员 B`、`仲裁员 C`、`访客`
   - 点击后 UI 变化：
     - 顶部状态栏的账号、短地址、角色标签立即变化。
     - 当前页面的业务操作面板重新计算权限。
     - 不弹出链上交易成功提示，因为切换账号不是链上交易。

2. **切换课堂场景**
   - 按钮：`重置 Demo`、`商品已创建`、`买家已付款`、`卖家已交付`、`买家申请退款`、`纠纷待押金`、`仲裁投票中`
   - 点击后 UI 变化：
     - 状态时间线跳到对应节点。
     - 商品摘要、资金流向、事件日志同步更新。
     - 页面显示一条说明：“已载入课堂演示场景，不代表真实链上交易。”

控制台不能和业务按钮混在一起。业务按钮必须仍然显示为 `purchaseItem()`、`markDelivered()`、`confirmReceived()`、`requestRefund()`、`openDispute()`、`respondDispute()`、`voteDispute()` 等合约动作。

## 3. 开场：Dashboard 总览页

### 3.1 初始画面

进入前端后首先展示 Dashboard。

UI 应显示：

- 顶部：项目名、当前账号、网络、交易状态。
- 左侧/中间：当前演示交易快照。
- 状态时间线：`Created -> Locked -> Delivered -> DisputeDepositPending -> Disputed -> Inactive`。
- 最近事件日志：`ItemCreated`、`ItemPurchased`、`ItemDelivered` 等。

讲解重点：

```text
这个前端不是完整二手平台，而是链上担保托管流程的演示控制台。
不同钱包地址会看到不同操作，合约权限由当前地址决定。
```

### 3.2 点击“连接钱包”

按钮：

```text
连接钱包
```

点击后 UI 变化：

1. 交易状态从 `Idle` 变为 `Pending`。
2. 顶部状态点出现加载或脉冲动画。
3. Toast 显示：“正在模拟 MetaMask 授权获取公开地址...”
4. 成功后默认切换到买家账号，状态变为 `Success`。
5. 顶部显示：`买家 Buyer`、短地址、网络状态。

讲解重点：

```text
这里用 Demo 账号模拟 MetaMask 多账号切换，后续接真实钱包时仍然按地址推导角色。
```

## 4. 正常交易流程

### 4.1 卖家创建商品

操作：

1. 在演示控制台点击 `卖家`。
2. 打开 `交易市场 Marketplace`。
3. 在创建商品表单输入：
   - 商品名：二手显示器
   - 价格：0.01 ETH
   - Metadata Hash：示例 hash
4. 点击业务按钮：

```text
创建商品 createItem()
```

点击后 UI 变化：

1. 按钮进入 `Pending`，显示“正在模拟 createItem() 上链...”。
2. 商品列表顶部新增一张商品卡片。
3. 商品状态标签显示 `Created / 可购买`。
4. 事件反馈新增：

```text
ItemCreated：卖家发布商品「二手显示器」
```

5. 该商品卡片在卖家账号下的购买按钮禁用，显示：

```text
不能购买自己发布的商品
```

讲解重点：

```text
卖家发布商品时需要锁定卖家保证金。商品进入 Created 状态，等待买家付款。
卖家不能购买自己的商品，这个限制既在合约里，也在前端权限提示里体现。
```

### 4.2 买家付款托管

操作：

1. 在演示控制台点击 `买家`。
2. Marketplace 商品卡片从卖家视角切换为买家视角。
3. 点击业务按钮：

```text
付款购买 purchaseItem()
```

点击后 UI 变化：

1. 按钮进入 `Pending`。
2. 商品卡片显示“正在锁入托管合约”。
3. 成功后商品状态从 `Created` 变为 `Locked / 托管中`。
4. 商品卡片上的购买按钮消失或禁用。
5. 事件日志新增：

```text
ItemPurchased：买家付款，0.01 ETH 已进入托管合约
```

6. 资金流向说明更新为：

```text
托管主金额：0.01 ETH
资金当前在 EscrowMarketplace 合约中，卖家暂时不能取走。
```

讲解重点：

```text
买家不是直接把钱转给卖家，而是转入智能合约托管。状态变成 Locked 后，商品信息被冻结。
```

### 4.3 卖家标记交付

操作：

1. 在演示控制台点击 `卖家`。
2. 打开 `交易详情 Trade Detail`。
3. 当前角色操作面板只显示卖家可用按钮。
4. 点击业务按钮：

```text
标记已交付 markDelivered()
```

点击后 UI 变化：

1. 交易状态从 `Locked` 变为 `Delivered`。
2. 状态时间线高亮 `Delivered` 节点。
3. 资金仍显示在托管合约中。
4. 事件日志新增：

```text
ItemDelivered：卖家已声明交付，确认收货倒计时开始
```

5. 买家操作折叠区提示：

```text
买家可确认收货或申请退款。
```

讲解重点：

```text
卖家声明交付后才开始买家确认窗口。这样避免卖家没交付也靠超时拿钱。
```

### 4.4 买家确认收货

操作：

1. 在演示控制台点击 `买家`。
2. 当前角色操作面板显示买家按钮。
3. 点击业务按钮：

```text
确认收货 confirmReceived()
```

点击后 UI 变化：

1. 交易状态从 `Delivered` 变为 `Inactive`。
2. 资金流向从“合约托管”变为“卖家可提现余额”。
3. 事件日志新增：

```text
ItemReceived：买家确认收货
TradeFinalized：主交易金额记入卖家 pendingWithdrawals
```

4. 页面显示结束状态：

```text
交易已完成，卖家可调用 withdrawProceeds() 提款。
```

讲解重点：

```text
合约采用 pull payment，不是在确认函数里直接转账给卖家，而是先记账到 pendingWithdrawals，再由卖家主动提款。
```

## 5. 协商退款流程

### 5.1 买家申请退款

操作：

1. 演示控制台点击 `买家`。
2. 场景切换到 `买家已付款` 或 `卖家已交付`。
3. 点击业务按钮：

```text
申请退款 requestRefund()
```

点击后 UI 变化：

1. 状态仍为 `Locked` 或 `Delivered`，但交易摘要出现退款标记。
2. 事件日志新增：

```text
RefundRequested：买家发起退款申请，等待卖家处理
```

3. 买家面板中 `确认收货 confirmReceived()` 禁用，原因：

```text
退款申请中，不能确认收货。
```

4. 卖家面板折叠说明变为：

```text
卖家可以同意退款 approveRefund()；退款申请存在时不能超时放款。
```

讲解重点：

```text
买家不能单方面退款，只能提出申请。是否退款需要卖家同意或进入仲裁。
```

### 5.2 卖家同意退款

操作：

1. 演示控制台点击 `卖家`。
2. 当前角色操作面板显示：

```text
同意退款 approveRefund()
```

3. 点击该按钮。

点击后 UI 变化：

1. 交易状态变为 `Inactive`。
2. 资金流向变为“买家可提现余额”。
3. 事件日志新增：

```text
RefundApproved：卖家同意退款
TradeFinalized：主交易金额记入买家 pendingWithdrawals
```

4. 页面结束提示：

```text
协商退款完成，买家可调用 withdrawProceeds() 提款。
```

讲解重点：

```text
这是无争议退款路径。卖家同意后，合约自动把资金记账给买家。
```

## 6. 纠纷仲裁流程

### 6.1 买家发起纠纷

操作：

1. 演示控制台点击 `买家`。
2. 场景切换到 `买家已付款` 或 `卖家已交付`。
3. 在证据输入框填写：

```text
证据哈希 / URI
```

4. 点击业务按钮：

```text
发起纠纷 openDispute()
```

点击后 UI 变化：

1. 状态从 `Locked` 或 `Delivered` 变为 `DisputeDepositPending`。
2. 状态时间线高亮 `DisputeDepositPending`。
3. 资金流向新增：
   - 托管主金额仍在合约。
   - 买家纠纷押金已提交。
   - 卖家纠纷押金待提交。
4. 事件日志新增：

```text
DisputeOpened：买家提交证据哈希并发起纠纷
```

5. 卖家折叠说明变为：

```text
当前需要卖家补交纠纷押金 respondDispute()，否则超时后发起方自动胜诉。
```

讲解重点：

```text
证据原文不上链，只把哈希上链，保证后续不能偷偷替换证据。
```

### 6.2 卖家补交纠纷押金

操作：

1. 演示控制台点击 `卖家`。
2. 当前角色操作面板显示：

```text
补交纠纷押金 respondDispute()
```

3. 点击该按钮。

点击后 UI 变化：

1. 状态从 `DisputeDepositPending` 变为 `Disputed`。
2. 状态时间线高亮 `Disputed`。
3. 资金流向显示：
   - 买家纠纷押金：已交
   - 卖家纠纷押金：已交
4. 页面出现“前往仲裁中心”按钮：

```text
前往仲裁中心
```

5. 事件日志新增：

```text
DisputeResponded：卖家补交纠纷押金，进入仲裁投票
```

讲解重点：

```text
双方都交押金后才进入仲裁。这样能防止一方随意发起纠纷却不承担成本。
```

### 6.3 仲裁员 A 质押并投票

操作：

1. 演示控制台点击 `仲裁员 A`。
2. 打开 `仲裁中心 Arbitration`。
3. 如果该仲裁员未质押，先点击：

```text
质押 0.1 ETH stakeAsArbiter()
```

4. 点击投票按钮之一：

```text
支持买家退款 voteDispute(true)
```

或：

```text
支持卖家放款 voteDispute(false)
```

点击后 UI 变化：

1. 仲裁员 A 行状态从“未投票”变为“已投票”。
2. 投票进度条增加 1 票。
3. 仲裁员 A 的投票按钮禁用，原因：

```text
同一仲裁员对本案只能投票一次。
```

4. 仲裁日志新增：

```text
ArbiterVoted：仲裁员 A 支持买家退款
```

讲解重点：

```text
仲裁员必须先质押保证金才有投票资格，并且每个仲裁员每笔纠纷只能投一次。
```

### 6.4 仲裁员 B 投出第二票，达成 2/3

操作：

1. 演示控制台点击 `仲裁员 B`。
2. 仲裁中心自动定位到仲裁员 B 席位。
3. 点击与仲裁员 A 相同方向的投票按钮：

```text
支持买家退款 voteDispute(true)
```

点击后 UI 变化：

1. 买家方向投票进度变为 `2 / 3`。
2. 裁决状态变为：

```text
裁决完成：支持买家退款
```

或若支持卖家：

```text
裁决完成：支持卖家放款
```

3. 所有投票按钮禁用。
4. 事件日志新增：

```text
DisputeResolved：达到 2/3 多数，合约执行裁决
```

5. 资金流向更新：
   - 买家胜诉：主交易金额记入买家可提现余额，卖家保证金被罚没或按规则处理。
   - 卖家胜诉：主交易金额记入卖家可提现余额。
   - 败诉方纠纷押金奖励多数票仲裁员。

讲解重点：

```text
3 个仲裁员中 2 个同向投票即可达成 2/3，多数达成后合约自动执行裁决。
```

### 6.5 仲裁员 C 访问已裁决案件

操作：

1. 演示控制台点击 `仲裁员 C`。
2. 打开仲裁中心。

UI 变化：

1. 仲裁员 C 行仍显示“未投票”。
2. 投票按钮禁用，原因：

```text
裁决已完成，无法继续投票。
```

3. 裁决卡片保持最终结果。

讲解重点：

```text
这说明裁决不是等所有人都投完，而是达到 2/3 多数后立即 finalized。
```

## 7. 访客只读视角

操作：

1. 演示控制台点击 `访客`。
2. 分别打开 Marketplace、Trade Detail、Arbitration。

UI 变化：

1. Marketplace：可以浏览商品，但创建和购买按钮禁用或要求连接有效买家账号。
2. Trade Detail：只显示交易摘要、状态时间线、资金流向、事件日志。
3. 当前角色操作面板显示：

```text
当前账号不是本交易的卖家、买家或仲裁员，仅可查看。
```

4. Arbitration：可以看投票进度和裁决结果，但投票按钮禁用，原因：

```text
当前账号不是有效仲裁员。
```

讲解重点：

```text
链上状态是公开可查的，但不是任何地址都能执行状态变更。
```

## 8. 建议的 8 分钟展示节奏

| 时间 | 内容 | 页面 |
|------|------|------|
| 0:00 - 0:40 | 项目定位：链上担保托管层，不是完整闲鱼 | Dashboard |
| 0:40 - 1:20 | 角色与账号：卖家、买家、仲裁员、访客 | Dashboard / 控制台 |
| 1:20 - 2:30 | 卖家创建商品，买家付款托管 | Marketplace |
| 2:30 - 3:30 | 卖家交付，买家确认，资金进入卖家可提现余额 | Trade Detail |
| 3:30 - 4:30 | 买家申请退款，卖家同意退款 | Trade Detail |

| 7:20 - 8:00 | Sepolia 部署证据与总结 | Deployment |

## 9. 展示时容易被问到的问题

### Q1：为什么不用一个下拉框选择“我是卖家/买家”？

回答：

```text
正式业务权限不应该由前端角色下拉框决定，而应该由当前钱包地址决定。
这里的 Demo 账号切换器是在模拟 MetaMask 多账号切换，底层仍然按地址推导权限。
```

### Q2：演示控制台会不会是作弊？

回答：

```text
控制台只负责课堂快速切换账号和状态，避免现场等待超时窗口或频繁重置数据。
真实业务按钮仍然按合约函数和角色权限展示，控制台不会伪装成链上功能。
```

### Q3：为什么访客也能看交易？

回答：

```text
区块链状态本身公开可查，访客能查看状态和事件日志；
但只有卖家、买家、有效仲裁员这些特定地址能发起状态变更。
```

### Q4：仲裁员是不是还是人在判断？

回答：

```text
是的，商品真假和线下交付无法完全链上判断。
本项目去中心化的是资金控制权和规则执行，不是事实判定本身。
仲裁的价值是把单点客服判断变成多签投票、证据哈希固定和事件可追溯。
```
