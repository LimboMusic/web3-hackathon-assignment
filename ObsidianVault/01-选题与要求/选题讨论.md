## 三、当前推荐

**基于双重签名的二手交易担保托管平台（Decentralized Escrow System）** — 最适合快速做出完整 Demo

### 推荐理由

- **真实痛点明确**：二手交易常见问题是“先付款怕被骗、先发货怕收不到钱”，担保托管能直接解释价值。
- **区块链必要性强**：资金进入智能合约后，平台或任意一方都不能随意挪用，释放和退款规则公开透明。
- **合约逻辑清晰**：交易状态可以设计为 `Created` → `Locked` → `Inactive`，非常适合课堂演示和测试。
- **前端演示直观**：左侧卖家创建商品，右侧买家付款和确认收货，切换 MetaMask 账户即可完整演示。
- **报告好写**：可以围绕“信任问题、托管机制、状态机、测试网部署、前后端交互”展开。

### 核心业务流程

1. 卖家创建商品，设置商品名称、描述、价格。
2. 买家调用 `pay()` 支付商品价格，资金进入合约托管，状态变为 `Locked`。
3. 卖家线下发货。
4. 买家收到货后调用 `confirmReceived()`，合约自动把资金转给卖家，状态变为 `Inactive`。
5. 如果买家付款后双方取消交易，可扩展 `refund()`，需要卖家同意后退款给买家。

### 推荐系统架构

- **智能合约**：Solidity + Hardhat，核心状态机、权限检查、事件日志、重入保护。
- **前端页面**：React/Vite + Ethers.js + MetaMask，做成单页面演示。
- **页面布局**：暗色系 + 玻璃拟物化风格，左侧“卖家面板”，右侧“买家面板”，中间显示交易状态时间线。
- **部署网络**：Sepolia 测试网。

### 合约接口草案

```solidity
createItem(string name, string description, uint256 price)
pay(uint256 itemId) payable
confirmReceived(uint256 itemId)
requestRefund(uint256 itemId)
approveRefund(uint256 itemId)
getItem(uint256 itemId) view
```

### 状态设计

| 状态 | 含义 | 可执行动作 |
|------|------|------------|
| `Created` | 卖家已创建商品，等待买家付款 | 买家 `pay()` |
| `Locked` | 买家已付款，资金在合约中托管 | 买家 `confirmReceived()` / 买家申请退款 |
| `Inactive` | 交易完成或退款完成 | 不可继续操作 |

### 演示脚本

1. 卖家账户连接 MetaMask，创建一个“二手显示器”商品，价格设为 `0.01 ETH`。
2. 切换买家账户，点击购买，MetaMask 确认支付。
3. 页面状态从 `Created` 变为 `Locked`，显示资金已托管。
4. 买家点击确认收货，合约把钱转给卖家。
5. 页面状态变为 `Inactive`，展示交易完成和事件记录。

### 测试用例

- 卖家可以创建商品。
- 买家付款金额必须等于商品价格。
- 卖家不能购买自己的商品。
- 未付款时不能确认收货。
- 只有买家可以确认收货。
- 确认收货后卖家收到资金。
- 完成后的交易不能重复确认或重复退款。
