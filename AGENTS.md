## 0. 必须优先遵守的开发顺序

后续所有开发、文档、报告、PPT、部署和重要修复都必须遵守：

```text
Obsidian 记录 -> 新建 OpenSpec change -> 开发实现 -> 回写 Obsidian 结果
```

具体要求：

- 没有先写 Obsidian 记录时，不要创建 OpenSpec change。
- 没有新的 OpenSpec change 时，不要开始修改合约、测试、前端、报告、PPT 或部署脚本。
- 每个重要改动都要有独立的 `openspec/changes/<change-name>/`。
- 完成实现、测试或部署后，必须把结果追加回 `ObsidianVault/04-开发记录/开发记录.md`。
- 这条顺序优先于后续所有开发便利性考虑；不要为了快而跳过。

- `04-开发记录/`：每日开发记录、问题和解决方案。

### OpenSpec

后续所有重要开发都必须使用 OpenSpec：

```text
openspec/
├── project.md
├── specs/
└── changes/
```

使用规则：

- `openspec/project.md`：项目背景、技术栈、工作流和全局约束。
- `openspec/specs/`：已经确认的项目规格。
- `openspec/changes/`：每次开发前创建一个新的 change，用于说明本次变更。
- 每个 change 至少包含 `proposal.md` 和 `tasks.md`。
- 涉及业务规则、状态机、角色权限、退款、仲裁、保证金、前端交互或部署流程变化时，必须补充对应 spec delta。
- OpenSpec 内容应优先使用中文；仅保留工具要求的结构关键词、命令、目录名、代码标识和专有名词英文。

### 代码目录

后续建议代码目录如下：

```text
contracts/      # Solidity 合约
scripts/        # 部署脚本
test/           # Hardhat 测试
frontend/       # React/Vite 前端
artifacts/      # Hardhat 生成文件，通常不手动编辑
```

如果使用 Hardhat 初始化项目，应优先遵循 Hardhat 默认结构。

## 3. 开发流程

### Step 1：确认 Obsidian 记录

实现前先检查：

- `ObsidianVault/02-方案设计/项目讨论结果-去中心化二手交易担保托管.md`
- `ObsidianVault/00-首页/项目首页.md`
- 相关主题文档，例如 `ObsidianVault/02-方案设计/OpenSpec 工作流.md`

如果需求发生变化，先更新 Obsidian 中的方案设计、开发记录或对应主题文档。

### Step 2：创建 OpenSpec change

在任何代码、报告或 PPT 实现前，先创建新的 OpenSpec change：

```text
openspec/changes/<change-name>/
├── proposal.md
├── tasks.md
└── specs/<capability>/spec.md  # 行为或需求变化时需要
```

要求：

- change 名称使用短横线，例如 `add-escrow-contract`、`add-refund-flow`。
- `proposal.md` 说明为什么改、改什么、影响范围。
- `tasks.md` 拆分可验证任务。
- OpenSpec change 创建完成后，才能开始实现。

### Step 3：实现合约

合约优先实现核心闭环：

1. 创建商品
2. 买家付款托管
3. 买家确认收货
4. 卖家收款
5. 买家申请退款
6. 卖家同意退款
7. 超时放款
8. 多签仲裁
9. 仲裁员保证金

合约必须关注：

- 状态机是否合法
- 权限检查是否完整
- 转账是否安全
- 是否防止重复放款或重复退款
- 事件是否足够前端展示和报告说明

### Step 4：编写测试

每个核心流程至少有一个测试：

- 卖家创建商品成功
- 买家付款金额必须正确
- 卖家不能购买自己的商品
- 买家确认收货后卖家收到资金
- 非买家不能确认收货
- 买家可以申请退款
- 卖家同意退款后买家收到资金
- 退款后不能重复操作
- 买家申请退款后不能超时放款
- 仲裁员质押保证金后才可投票
- 2/3 仲裁员同意后执行裁决

测试结果需要记录到：

```text
ObsidianVault/04-开发记录/开发记录.md
```

### Step 5：实现前端

前端建议使用：

- React
- Vite
- Ethers.js
- MetaMask

前端必须能展示：

- 当前钱包地址
- 当前网络
- 商品信息
- 当前交易状态
- 卖家面板
- 买家面板
- 仲裁员面板
- 事件记录或操作日志

不要把前端做成纯介绍页。第一屏应该是可交互的 Demo 页面。

### Step 6：部署测试网

部署到 Sepolia 后，需要记录：

- 合约地址
- 部署账户
- 部署网络
- 部署交易哈希
- 部署时间
- 重要测试交易哈希

记录位置：


## 5. 合约设计约束

推荐状态机：

```solidity
enum State {
    Created,
    Locked,
    Disputed,
    Inactive
}
```

状态含义：

- `Created`：商品已创建，等待买家付款。
- `Locked`：买家已付款，资金托管中。
- `Disputed`：交易进入纠纷，等待仲裁。
- `Inactive`：交易结束，已放款或已退款。

基础规则：

- 卖家不能购买自己的商品。
- 买家付款金额必须等于商品价格。
- 只有买家可以确认收货。
- 买家确认收货后，资金释放给卖家。
- 买家不能单方面退款。
- 卖家同意退款后，资金退给买家。
- 买家申请退款后，不允许卖家直接超时放款。
- 纠纷状态下，由仲裁员投票决定退款或放款。
- 每个仲裁员每笔纠纷只能投票一次。
- 达到 2/3 多数后，合约自动执行裁决。
- 仲裁员需要质押保证金后才有投票资格。

## 6. 前端设计约束

前端应服务于课堂演示，不做过度复杂的平台系统。

推荐布局：

- 顶部：项目名称、钱包连接、网络状态。
- 左侧：卖家面板。
- 中间：商品卡片、状态时间线、事件日志。
- 右侧：买家面板。
- 底部或独立区域：仲裁员面板。

交互必须清楚体现：

- 买家付款后，钱进入合约托管。
- 状态从 `Created` 变为 `Locked`。
- 买家确认收货后，状态变为 `Inactive`。
- 退款不能由买家单方面完成。
- 纠纷需要仲裁员投票。

## 9. 安全注意事项

严禁提交：

- 私钥
- 助记词
- 真实钱包敏感信息
- Alchemy / Infura 私密 API Key
- `.env` 中的真实配置

智能合约涉及转账时必须注意：

- 先更新状态，再转账。
- 避免重复提款。
- 避免重入攻击。
- 对每个外部可调用函数做权限检查。
- 对每个状态变化做状态检查。
