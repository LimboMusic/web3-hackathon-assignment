# 提案：实现核心担保托管合约

## 背景

[[技术实现文档]] 已将项目拆为 11 个顺序步骤，第 1 步负责 Hardhat 工程骨架，第 2 步开始进入核心业务合约。本 change 对应 `add-escrow-core-contract`，目标是在不引入退款、超时、多签仲裁等后续复杂逻辑的前提下，先跑通“卖家上架商品 -> 买家付款托管 -> 卖家标记交付 -> 买家确认收货 -> 卖家提款”的最小闭环。

该闭环是课程 Demo 的基础：它需要清楚体现资金不是直接打给卖家，而是先进入智能合约托管；买家确认收货后，合约才把资金释放给卖家。

## 变更内容

- 新增核心合约 `EscrowMarketplace`，包含商品结构、状态机、商品编号、自增存储和卖家可提现余额。
- 实现基础状态：`Created`、`Locked`、`Delivered`、`DisputeDepositPending`、`Disputed`、`Inactive`。
- 实现卖家操作：
  - 创建商品；
  - 在未被购买前修改价格和元数据哈希；
  - 在未被购买前下架商品；
  - 在买家付款后标记已交付；
  - 在买家确认收货后通过 pull payment 提款。
- 实现买家操作：
  - 按商品价格付款并进入托管；
  - 在卖家标记交付后确认收货；
  - 付款后冻结核心交易信息，避免卖家篡改价格或商品描述影响交易。
- 实现基础安全边界：
  - 卖家不能购买自己的商品；
  - 买家付款金额必须等于商品价格；
  - 下架商品不能购买；
  - 已购买商品不能重复购买；
  - 非买家不能确认收货；
  - 非卖家不能标记交付或提款；
  - 状态更新和余额记账先于外部转账；
  - 使用重入保护保护付款、确认和提款路径。
- 为关键动作发出事件，便于后续前端时间线、事件日志和报告说明。
- 编写核心单元测试，覆盖主流程、金额校验、权限校验、重复操作和 pull 提款。

- 测试：`test/EscrowMarketplace.*`
- 文档：`ObsidianVault/02-方案设计/技术实现文档.md`、`ObsidianVault/04-开发记录/开发记录.md`

## 非目标

- 不实现买家申请退款、卖家同意退款、双方确认退款。
- 不实现交付超时、确认超时或超时放款。
- 不实现仲裁员质押、仲裁员退出、当事人回避。
- 不实现纠纷押金、证据哈希、多签投票或裁决执行。
- 不实现卖家保证金、举报和信誉计数。
- 不接入 Sepolia 部署，不创建前端页面。
