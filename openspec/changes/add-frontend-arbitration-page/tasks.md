# 任务清单

## 0. 前置文档

- [ ] 阅读 `ObsidianVault/02-方案设计/UI设计稿/arbitration.html`，记录仲裁治理台、投票面板、席位管理、仲裁员表格和日志布局。
- [ ] 阅读 `openspec/specs/escrow-core/spec.md` 中仲裁员质押、纠纷押金、投票阈值和裁决规则。
- [ ] 在 `ObsidianVault/04-开发记录/开发记录.md` 中记录开始实现 Arbitration 页面骨架。

## 1. 页面结构

- [ ] 在 `frontend/src/pages/Arbitration.tsx` 新增仲裁中心页面。
- [ ] 页面顶部展示当前纠纷订单、合约状态 `Disputed`、证据哈希摘要、纠纷押金状态和投票截止提示。
- [ ] 展示投票进度卡片，至少包含支持买家退款票数、支持卖家放款票数、所需 2/3 阈值和当前裁决状态。
- [ ] 展示仲裁员席位管理卡片，包含当前地址质押状态、质押金额、退出限制说明和模拟质押/退出按钮。
- [ ] 展示仲裁员明细表，列出地址摘要、保证金质押状态、投票方向、合流状态。
- [ ] 展示仲裁日志，包含投票提交和裁决完成事件。

## 2. Mock 仲裁数据与投票

- [ ] 建立 mock 仲裁员列表，至少 3 名有效仲裁员。
- [ ] 为仲裁员、投票方向、纠纷状态和仲裁日志定义 TypeScript 类型或接口。
- [ ] 每个仲裁员必须有 `address`、`staked`、`locked`、`hasVoted`、`vote` 字段或等价结构。
- [ ] 投票按钮支持模拟“支持买家退款”和“支持卖家放款”。
- [ ] 同一 mock 仲裁员对同一纠纷只能投一次，重复投票必须显示错误提示。
- [ ] 达到 2/3 阈值后，页面必须进入最终裁决状态并禁用继续投票。
- [ ] 裁决完成后事件日志追加 `DisputeResolved` 或设计稿中的等价仲裁完成事件。

## 3. 规则表达

- [ ] 页面必须明确展示“仲裁员需先质押保证金才可投票”。
- [ ] 页面必须明确展示“买卖双方不可作为自己交易的仲裁员”的回避规则。
- [ ] 页面必须明确展示“败诉方纠纷押金奖励多数票仲裁员”的资金规则，但本步只做说明和 mock 结果。
- [ ] 页面必须区分多数票、少数票、未投票仲裁员的奖励状态表达。

## 4. 验证

- [ ] 运行 `rtk npm run build`。
- [ ] 手工演示 2/3 买家胜诉路径，确认票数、裁决状态、日志同步更新。
- [ ] 手工演示 2/3 卖家胜诉路径，确认票数、裁决状态、日志同步更新。
- [ ] 手工检查重复投票、未质押投票和裁决后投票都有错误或禁用反馈。
- [ ] 检查仲裁员表格在窄屏下不横向挤爆主要内容，可滚动或合理换行。
- [ ] 运行 `rtk npx openspec validate add-frontend-arbitration-page --strict`。
